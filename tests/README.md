# Playwright QA Automation — Agency X Grant Portal

Production-grade [Playwright](https://playwright.dev/) framework that validates
the public **Grant Application** form (LWC `grantApplicationForm`) on the Agency
X Experience Cloud site — across three browsers, with accessibility and visual
regression coverage and rich [Allure](https://allurereport.org/) reporting.

## Architecture

The suite follows the **Page Object Model** with custom Playwright fixtures, so
specs read as business scenarios rather than low-level automation.

```
tests/
├── e2e/
│   ├── grant-application.spec.js   # Functional & negative scenarios
│   └── accessibility.spec.js       # Automated WCAG 2.1 A/AA audit (axe-core)
├── visual/
│   ├── grant-application.visual.spec.js        # Visual-regression scenario
│   └── *-snapshots/                            # Committed baselines (per OS)
├── pages/
│   └── grant-application.page.js   # Page Object: locators + actions
├── fixtures/
│   └── portal.fixtures.js          # Custom fixture: pre-opened application page
├── data/
│   └── applicants.js               # Test-data builders (randomised, idempotent)
├── global-setup.js                 # One-time authentication → storageState
└── .auth/state.json                # Cached session (git-ignored, auto-generated)
```

### Why this design

- **Page Object Model** — locators and actions live in one place; specs never
  touch raw selectors, so UI changes require a single edit.
- **Custom fixtures** — every test receives an already-opened, authenticated
  `grantApplicationPage`; no repeated boilerplate.
- **Authenticate once** — `global-setup.js` signs in via the Salesforce CLI and
  saves the session to `storageState`. Tests run fully in parallel and share it,
  which keeps the suite fast and avoids rate limiting.
- **Idempotent data** — phone numbers (the org's unique key) are randomised, so
  the suite can run repeatedly without cleanup.
- **Cross-browser** — functional and accessibility specs run on Chromium,
  Firefox and WebKit; visual regression is pinned to one engine for stability.

## Test types

| Type              | What it checks                                                      |
| ----------------- | ------------------------------------------------------------------- |
| Functional (E2E)  | Form rendering, eligible/not-eligible submissions, support options  |
| Negative          | Invalid Singapore phone number and postal code are rejected         |
| Accessibility     | WCAG 2.1 A/AA scan via `@axe-core/playwright` (no critical/serious) |
| Visual regression | Pristine form matches an approved pixel baseline                    |

## Prerequisites

1. Node.js 18+ and project dependencies installed:

```bash
npm install
npx playwright install
```

2. An authenticated Salesforce org (Salesforce CLI):

```bash
sf org login web --alias grant-portal
```

3. A `.env` file in the project root:

```bash
PORTAL_URL=https://<your-domain>.my.site.com/agencyxgrants/
SF_TARGET_ORG=<username-or-alias>
SF_SITE_ID=<Experience-site-id>
SF_COMMUNITY_PATH=<community-url-path>
```

## Running the tests

```bash
npm run test:e2e                       # all browsers + a11y + visual (headless)
npm run test:e2e -- --project=chromium # a single browser
npm run test:e2e -- --project=visual   # visual regression only
npm run test:e2e:update-snapshots      # refresh visual baselines (this OS)
npm run test:e2e:headed                # watch it run in a real browser
npm run test:e2e:ui                    # Playwright interactive UI mode
npm run test:e2e:report                # open the last Playwright HTML report
```

> Visual baselines are OS-specific. Local runs use your OS baseline; the Linux
> baselines used by CI are generated by the **Update Visual Baselines** workflow
> (Actions → run manually) and committed automatically.

## Running in Docker

For a fully reproducible run (no local Node/browser setup), use the containerised
runner — see the [root README](../README.md) or run directly:

```bash
export SFDX_AUTH_URL="$(sf org display --verbose --target-org <org> \
  --json | python3 -c 'import sys,json;print(json.load(sys.stdin)["result"]["sfdxAuthUrl"])')"
docker compose run --rm e2e
```

## Allure reporting

```bash
npm run allure:serve        # generate + open a temporary report (quickest)
npm run allure:generate     # build static report into allure-report/
npm run allure:open         # open the generated static report
npm run test:e2e:allure     # run tests, then build & open the report
```

The Allure report includes, per test:

- **Suites / features / stories** grouped via `epic` / `feature` / `story`
- **Severity** and **tags** (`smoke`, `regression`, `validation`, `a11y`, …)
- **Step-by-step** breakdown from `test.step(...)`
- **Screenshots** attached at each key stage
- **Trace, video and screenshots on failure** (from the Playwright config)

## CI/CD

`.github/workflows/e2e-tests.yml` runs on every push and pull request:

1. **LWC Unit Tests** — Jest with coverage.
2. **E2E matrix** — one job per browser (`chromium`, `firefox`, `webkit`); each
   authenticates to the org, runs the functional/a11y suites and uploads its
   Allure results as an artifact.
3. **Visual Regression** — a dedicated Chromium job compares against the Linux
   baseline.
4. **Publish Allure Report** — merges results from all browsers, restores
   historical trends and publishes one report to **GitHub Pages**.

`.github/workflows/update-visual-baselines.yml` regenerates the Linux visual
baselines on demand and commits them back.

## Test coverage

Every functional/accessibility scenario runs on **Chromium, Firefox and
WebKit**; the visual-regression scenario is pinned to a single engine.

| Scenario                                      | Severity | Tags                    |
| --------------------------------------------- | -------- | ----------------------- |
| Application form renders all fields           | critical | smoke, ui               |
| Eligible applicant submits successfully       | blocker  | regression, happy-path  |
| Applicant above income threshold not eligible | critical | regression, eligibility |
| Eligible submission with Option Two           | normal   | regression, happy-path  |
| Eligible submission with Option Three         | normal   | regression, happy-path  |
| Invalid Singapore phone number rejected       | normal   | regression, validation  |
| Invalid Singapore postal code rejected        | normal   | regression, validation  |
| Form meets WCAG 2.1 A/AA (no critical issues) | critical | a11y, wcag              |
| Form matches its approved visual baseline     | normal   | visual, ui              |
